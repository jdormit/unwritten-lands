# Unwritten Lands — Implementation Prompt

## Overview

Build **Unwritten Lands**, a browser-based narrative strategy game inspired by King of Dragon Pass. The player leads a clan through years of events, making decisions that affect resources, relationships, and storylines. All narrative content — events, dialog, world lore — is generated by an LLM at runtime through a two-call architecture (Director + Narrator). The game ends after 10 in-game years with a climactic crisis shaped by the player's accumulated decisions.

This is a prototype. Prioritize getting the core loop working over polish. Text-heavy is fine — this is fundamentally an interactive fiction engine with resource management.

---

## Tech Stack

- **Vite + React + TypeScript**
- **Vercel AI SDK** for LLM calls, using this backend: https://github.com/jdormit/ai-sdk-codex-oauth
- **Tailwind CSS** for styling
- **React state** (useReducer) for game state management
- **IndexedDB or localStorage** for save/resume persistence (must support saving and resuming a single game across browser sessions)
- **No game engine.** This is a UI app, not a canvas game. All "rendering" is React components.

## Game Design

### Core Loop

```
1. WORLD GENERATION (game start)
   - LLM generates: setting, geography, mythology, 3 neighboring clans,
     3 advisors, and a looming existential threat that will arrive in Year 10

2. SACRED TIME (start of each year)
   - Brief recap of the previous year (LLM-generated)
   - Player allocates a small number of magic points across priorities
     (war / harvest / diplomacy — 3 categories, ~3-5 points to distribute)

3. EACH SEASON (4 seasons per year = 4 events per year)
   - An event fires (generated by Director → Narrator pipeline)
   - 3 advisors each give a one-line opinion
   - Player picks from 3 options
   - Resources update, relationships change, storyline flags update
   - Optional: between seasons, player can initiate one action
     (raid, trade, explore, hold feast, etc.) which generates its own
     mini-event through the same pipeline

4. YEAR 10: CLIMAX
   - The looming threat arrives as a final multi-part event
   - Outcome shaped by: resource levels, alliances, resolved storylines,
     accumulated history
   - LLM generates an epilogue saga summarizing the clan's full story

5. EARLY LOSS
   - If any resource hits 0, the clan collapses
   - LLM generates a collapse narrative and abbreviated epilogue
```

### Resources

Four resources, each on a 0-10 scale:

| Resource | Represents | Hits 0 |
|----------|-----------|--------|
| **People** | Population, health, morale | Clan scatters or dies out |
| **Wealth** | Food, cattle, goods, material prosperity | Starvation and destitution |
| **Magic** | Spiritual favor, connection to gods/spirits | Cursed, abandoned by the gods |
| **Reputation** | How neighboring clans perceive you | Isolated, raided, no allies |

Starting values: all at 5 (or slightly varied based on world generation choices).

### Resource Flow Design Principle

Every event option should touch at least two resources, ideally in opposite directions. The player should almost never get a "free" gain. Common tradeoff axes:

- **Wealth ↔ People** (push people harder for wealth, or spend wealth on morale)
- **Wealth ↔ Magic** (sacrifice goods to the gods, or exploit sacred resources)
- **Wealth ↔ Reputation** (hard bargains vs. generous gifts)
- **People ↔ Reputation** (war costs lives but earns respect; refugees gain people but may anger neighbors)
- **Magic ↔ People** (strict religious observance vs. letting people rest)
- **Magic ↔ Reputation** (public miracles spend magic but awe neighbors)

Resources are never spent through a menu. They change through narrative choices only. The player experiences a story decision, not a spreadsheet.

### Advisors

Three advisors, generated during world creation, each with:

- A name
- An archetype (warrior, mystic, diplomat — or similar)
- A personality (aggressive, cautious, pragmatic, pious, etc.)
- A patron god/spirit from the generated mythology

Each advisor gives a one-line opinion on every event. Their advice should:

- Reflect their personality consistently
- Sometimes disagree with each other
- Occasionally surprise (the warrior counsels restraint when People is critically low)
- Reference the game state and active storylines when relevant

### Neighboring Clans

3-4 clans generated at world start, each with:

- A name
- A personality/reputation (warlike, peaceful traders, reclusive mystics, treacherous schemers)
- A relationship score with the player (-3 to +3, starting at varied values)
- Brief cultural details that make them feel distinct

Clan relationships shift based on player decisions. Clans can be allies, enemies, or neutral. They take actions that generate events (raids, trade offers, requests for help, marriage proposals, etc.).

### Storylines

The Director tracks active storylines as a list of flags/descriptions. Rules:

- Max 4-5 active storylines at once
- Each storyline has a state: introduced → escalating → climax → resolved
- Storylines should typically resolve within 3-6 events of being introduced
- Resolved storylines get added to permanent history and may trigger new ones
- The looming threat is a special permanent storyline that escalates on a fixed schedule:
  - Years 1-3: distant foreshadowing
  - Years 4-6: signs intensify, preparation becomes urgent
  - Years 7-9: imminent, forces hard choices
  - Year 10: climax arrives

### Pacing / The Looming Threat

At world generation, the LLM creates an existential threat tied to the mythology (e.g., "The Ember Wyrm stirs beneath the mountain," "A rival kingdom builds a war fleet," "A prophesied eclipse will shatter the barrier between worlds"). This threat:

- Cannot be prevented, only prepared for
- Gets woven into events with increasing frequency as the game progresses
- Creates the dramatic arc that prevents the game from feeling aimless
- The Year 10 climax resolution depends on accumulated resources, alliances, resolved storylines, and specific preparations the player made

## LLM Architecture

### Two-Call Pipeline: Director → Narrator

Each event requires two sequential LLM calls.

#### Call 1: The Director

**Purpose:** Decide what happens structurally. Output is strict JSON, no prose.

**Input:** Full game state (see Game State Schema below)

**Output schema:**

```typescript
interface DirectorOutput {
  event_type: string; // e.g. "raid", "trade", "omen", "internal_dispute", "supernatural", "diplomatic"
  involves_clans: string[]; // names of any neighboring clans involved
  storylines_referenced: string[]; // IDs of active storylines this advances
  new_storyline?: { // if this event starts a new storyline
    id: string;
    description: string;
    state: "introduced";
  };
  storyline_state_changes?: { // if this advances/resolves existing storylines
    id: string;
    new_state: "escalating" | "climax" | "resolved";
  }[];
  season_context: string; // how the current season affects this event
  threat_foreshadowing?: string; // if this event connects to the looming threat
  options: [DirectorOption, DirectorOption, DirectorOption]; // always exactly 3
}

interface DirectorOption {
  summary: string; // brief description of the choice
  tone: "cautious" | "bold" | "diplomatic" | "ruthless" | "pious" | "pragmatic";
  resource_effects: {
    people?: number; // -2 to +2
    wealth?: number;
    magic?: number;
    reputation?: number;
  };
  relationship_effects?: {
    clan_name: string;
    change: number; // -2 to +2
  }[];
  flags_added?: string[];
  flags_removed?: string[];
}
```

**Director system prompt must include these rules:**

Event selection:
- If any resource is ≤ 2, generate a crisis event related to that resource (but always offer at least one option that could help)
- If all resources are ≥ 5, introduce a new storyline or escalate an existing one
- Reference active storylines ~50% of the time
- Season matters: winter = scarcity/spiritual unease, spring = planting/new beginnings/raids resume, summer = trade/war/festivals, autumn = harvest/preparation
- Don't repeat event types back to back
- In years 7+, increase frequency of threat-related events

Option design:
- Always exactly 3 options
- At least one "cautious" option with small or no resource changes
- At least one "bold" option with bigger swings (+2/-1 or similar)
- At least one option involving a tradeoff between two different resources
- No single option should change any resource by more than ±2
- No option should change more than 2 resources
- Options should not be symmetrical — some should be genuinely better or worse depending on situation

#### Call 2: The Narrator

**Purpose:** Write the prose that the player sees. Also strict JSON output.

**Input:** Director output + full game state

**Output schema:**

```typescript
interface NarratorOutput {
  event_narrative: string; // 2-4 sentences describing the situation
  advisor_opinions: {
    advisor_name: string;
    opinion: string; // 1-2 sentences, in character
  }[];
  option_texts: string[]; // exactly 3, each a short imperative sentence
  // e.g. "Sell only the wool." / "Demand they owe you a favor."
}
```

**Narrator system prompt must include:**

- Voice: saga-like but not overwrought. Concrete sensory details. Short paragraphs.
- Advisors speak in distinct voices reflecting their personality
- Advisor opinions should reflect current game state (not just personality defaults)
- Option text is concise — imperative sentences, 1 line each
- Don't reveal resource numbers in the narrative text (UI shows those separately)
- Reference established world details, mythology, and recent history for continuity

### World Generation

A single LLM call at game start. Output schema:

```typescript
interface WorldGeneration {
  setting: {
    name: string; // name of the region
    geography: string; // 2-3 sentences
    tone: string; // the mythic/cultural flavor
  };
  mythology: {
    pantheon_summary: string; // 2-3 sentences about the gods/spirits
    creation_myth: string; // 1-2 sentences
    key_deities: { name: string; domain: string; personality: string }[];
  };
  clan: {
    name: string;
    backstory: string; // 2-3 sentences about the clan's history and migration
  };
  advisors: {
    name: string;
    archetype: "warrior" | "mystic" | "diplomat";
    personality: string; // 2-3 adjectives
    patron_deity: string;
    speaking_style: string; // brief description of how they talk
  }[];
  neighboring_clans: {
    name: string;
    reputation: string; // e.g. "warlike raiders", "wealthy traders"
    personality: string;
    initial_relationship: number; // -2 to +2
    detail: string; // one distinguishing cultural detail
  }[];
  looming_threat: {
    name: string;
    description: string; // 2-3 sentences
    nature: string; // "military" | "supernatural" | "natural" | "political"
    foreshadowing_hints: string[]; // 3-4 ways it can be hinted at early
    preparation_axes: string[]; // what kinds of preparation help against it
    // e.g. ["military strength", "magical protection", "allied clans"]
  };
}
```

The world generation prompt should instruct the LLM to create a setting that feels mythic and culturally specific (not generic fantasy), with internal consistency between the mythology, the clans, and the threat.

### Sacred Time / Year Recap

A simpler LLM call at the start of each year:

```typescript
interface SacredTimeOutput {
  year_recap: string; // 2-3 sentences summarizing last year
  omens: string; // 1-2 sentences of prophecy/foreshadowing for the coming year
  threat_status: string; // 1 sentence on the looming threat's progression
}
```

### Epilogue Generation

At game end (victory or collapse), one final LLM call receives the complete game history and generates:

```typescript
interface EpilogueOutput {
  saga_title: string;
  saga_text: string; // a narrative summary of the clan's entire story,
  // referencing key decisions, alliances, betrayals, losses, and triumphs
  // Should read like an oral history or chronicle
  // 3-6 paragraphs
  outcome: "triumph" | "survival" | "pyrrhic_victory" | "defeat" | "collapse";
}
```

## Game State Schema

The complete state passed to each LLM call:

```typescript
interface GameState {
  world: WorldGeneration; // from initial generation
  
  resources: {
    people: number; // 0-10
    wealth: number;
    magic: number;
    reputation: number;
  };
  
  clan_relationships: {
    clan_name: string;
    score: number; // -3 to +3
  }[];
  
  active_storylines: {
    id: string;
    description: string;
    state: "introduced" | "escalating" | "climax" | "resolved";
    started_year: number;
    started_season: string;
  }[];
  
  resolved_storylines: {
    id: string;
    description: string;
    resolution: string;
    year_resolved: number;
  }[];
  
  current_year: number; // 1-10
  current_season: "spring" | "summer" | "autumn" | "winter";
  
  magic_allocation: { // set during Sacred Time
    war: number;
    harvest: number;
    diplomacy: number;
  };
  
  flags: string[]; // miscellaneous state flags
  // e.g. "elk_clan_owes_favor", "iron_shortage", "blessed_by_storm_god"
  
  event_history: { // last ~8 events for context window management
    year: number;
    season: string;
    summary: string; // 1-2 sentence summary of what happened and what the player chose
  }[];
  
  full_history: { // complete log for epilogue generation
    year: number;
    season: string;
    summary: string;
  }[];
}
```

Note: `event_history` is a rolling window of recent events passed to Director/Narrator calls for immediate context. `full_history` is the complete log used only for epilogue generation. Keep `event_history` to ~8 entries to manage prompt size.

## UI Design

### Layout

Aim for a warm, parchment-like aesthetic. Think muted earth tones, serif fonts for narrative text, simple iconography for resources.

```
┌──────────────────────────────────────────────────┐
│  [Clan Name]          Year 3, Summer             │
│  People: ████████░░  Wealth: ██████░░░░          │
│  Magic:  ███████░░░  Reputation: █████░░░░░      │
├──────────────────────────────────────────────────┤
│                                                  │
│  [Event narrative text here, 2-4 sentences]      │
│                                                  │
│  ┌─────────────────────────────────────────────┐ │
│  │ Harka: "Keep the iron. We'll need it."      │ │
│  │ Mirela: "The spirits say to be generous."   │ │
│  │ Tomas: "Demand a favor in return."          │ │
│  └─────────────────────────────────────────────┘ │
│                                                  │
│  [Option 1 button]                               │
│  [Option 2 button]                               │
│  [Option 3 button]                               │
│                                                  │
├──────────────────────────────────────────────────┤
│  [History log / sidebar - scrollable]            │
│  Recent events listed chronologically            │
└──────────────────────────────────────────────────┘
```

### Screens

1. **New Game** — brief intro, then world generation (show a loading state while the LLM works, maybe with thematic flavor text)
2. **Sacred Time** — year recap, omens, magic allocation sliders
3. **Event** — the main game screen (layout above)
4. **Epilogue** — the saga text, displayed nicely, with a "New Game" option
5. **Game Over** — collapse narrative, then abbreviated epilogue

### UX Details

- After selecting an option, show the resource changes with brief animation (bars shifting, maybe +/- indicators)
- History log should be accessible but not dominant — maybe a collapsible sidebar or a tab
- Advisor opinions should feel like speech — maybe style them differently from narrative text (italics, or speech bubble styling)
- Loading states during LLM calls need to feel thematic, not broken — use "The spirits deliberate..." or "The wind carries whispers..." flavor text
- Save happens automatically after each event resolution

### Responsive Design

Should work on both desktop and mobile. The event + options + advisors layout should stack vertically on narrow screens.

## Implementation Notes

### Vercel AI SDK Usage

Use the Vercel AI SDK's `generateObject` (or equivalent structured output method) for all LLM calls to get reliable JSON parsing. The backend at https://github.com/jdormit/ai-sdk-codex-oauth handles auth and model routing.

Both the Director and Narrator calls should use structured output / JSON schema validation to ensure reliable parsing. Don't rely on parsing freeform text.

### Context Window Management

The full game state (world generation + current state + recent history) needs to fit comfortably in the context window alongside system prompts. Rough budget:

- World generation data: ~500-800 tokens
- Current state (resources, relationships, storylines, flags): ~200-400 tokens
- Recent event history (8 events): ~400-600 tokens
- System prompt: ~500-1000 tokens
- Room for output: ~500-1000 tokens

This should fit well within standard model context windows. If it gets tight, compress the world generation data after initial creation (keep only what's needed for ongoing events).

### Error Handling

- If an LLM call fails, retry once, then show an in-theme error ("The spirits are silent... try again")
- Validate Director output before passing to Narrator (resource effects within bounds, correct number of options)
- If validation fails, retry the Director call with a note about what was wrong

### Persistence

Save the full GameState to IndexedDB (or localStorage if simpler) after every event resolution. On app load, check for a saved game and offer to resume or start new. Only one save slot needed for the prototype.

## What to Build First

Suggested implementation order:

1. **Game state types and reducer** — define the TypeScript interfaces above and the state management logic
2. **World generation** — the initial LLM call and display of the generated world
3. **Event pipeline** — Director → Narrator → display → player choice → state update. This is the core loop. Get one event working end to end.
4. **Sacred Time** — year transition, recap, magic allocation
5. **Season progression** — wire up the 4-seasons-per-year cycle with events
6. **Storyline tracking** — active storylines, advancement, resolution
7. **Endgame** — Year 10 climax + epilogue generation
8. **Early loss** — resource-hits-zero detection and collapse narrative
9. **Persistence** — save/load
10. **Polish** — loading states, animations, responsive design, history log

Get through step 5 and you have a playable game. Everything after that is enrichment.
